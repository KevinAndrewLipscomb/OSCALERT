000-README

Abstract

This document describes additional steps that you must perform, after checking out the OSCALERT project from its repository, to achieve a clean compilation or to execute the application. This document does not cover how to meet other prerequisites, such as setting up an ASP.NET server, creating and populating the database schema, and setting up a cron daemon.

1 Recommended methods

The use of NTFS reparse points is recommended in this document because it allows Subversion operations to take place without undesirable impact to project environments, and without violating Subversion rules. If you simply exported OSCALERT from its repository and you do not intend to perform subsequent updates or any other Subversion operations on it (not recommended), you can simply make copies instead of NTFS reparse points.

2 Establish hooks to adjunct projects

2.1 Hook up the KiUsercontrolFoundry project

OSCALERT uses UserControls from the KiUsercontrolFoundry adjunct project. The KiUsercontrolFoundry is used by multiple production-eligible projects and is version controlled independently from those production-eligible projects. To hook up the KiUsercontrolFoundry project, do the following:
  1. Check out the https://kveo-it-project.svn.sourceforge.net/svnroot/kveo-it-project/trunk/foundry/usercontrol/ki project. Put it in a KiUsercontrolFoundry folder. Make sure the KiUsercontrolFoundry folder is not inside the OSCALERT folder tree.
  2. Create a junction reparse point named ki inside the OSCALERT/usercontrol folder that points to the KiUsercontrolFoundry/usercontrol/ki folder. You should end up with two folders inside the OSCALERT/usercontrol folder: app and ki. You can make junction points with Hermann Schinagl’s Link Shell Extension tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OSCALERT/usercontrol
    mklink /j ki KiUserControlFoundry/usercontrol/ki

2.2 Hook up the KiDotnetFoundation project (only if compiling)

OSCALERT uses object code from the KiDotnetFoundation adjunct project. The KiDotnetFoundation is used by multiple production-eligible projects and its source code is version controlled independently from those production-eligible projects. The object code is already provided as OSCALERT/bin/ki.dll to enable execution of OSCALERT in a production environment, but to hook up the KiDotnetFoundation project for the sake of a clean compile of OSCALERT, do the following:
  1. Check out the https://kveo-it-project.svn.sourceforge.net/svnroot/kveo-it-project/trunk/foundation/dotnet/ki project. Put it in a KiDotnetFoundation folder. Make sure the KiDotnetFoundation folder is not inside the OSCALERT folder tree.
  2. Create a junction reparse point named ki inside the OSCALERT/lib folder that points to the KiDotnetFoundation folder. You should end up with a folder inside the OSCALERT/lib folder called ki. You can make junction points with Hermann Schinagl’s Link Shell Extension tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OSCALERT/lib
    mklink /j ki KiDotnetFoundation

2.3 Hook up the oscalib project (only if compiling)

OSCALERT uses object code from the oscalib adjunct project. The oscalib is used by multiple production-eligible projects and its source code is version controlled independently from those production-eligible projects. The object code is already provided as OSCALERT/bin/oscalib.dll to enable execution of OSCALERT in a production environment, but to hook up the oscalib project for the sake of a clean compile of OSCALERT, do the following:
  1. Check out the https://svn.code.sf.net/p/kveo-it-project/code/trunk/oscalib project. Put it in a oscalib folder. Make sure the oscalib folder is not inside the OSCALERT folder tree.
  2. Create a junction reparse point named oscalib inside the OSCALERT/lib folder that points to the oscalib folder. You should end up with a folder inside the OSCALERT/lib folder called oscalib. You can make junction points with Hermann Schinagl’s Link Shell Extension tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OSCALERT/lib
    mklink /j oscalib oscalib

3 Put appropriate config files in the application root folder

OSCALERT will not run without Web.config and unpublished.config files in its top-level folder. Because the contents of those files are environment-specific, and we wish to keep the canonical ASP.NET parts of the OSCALERT folder tree environment-neutral, those config files are kept in a non-ASP.NET-canonical OSCALERT/config subfolder tree. To put config files that are appropriate for your environment into the top-level OSCALERT folder, do the following:
  1. If there is no unpublished.config file inside the OSCALERT/config folder tree (this should be true for projects kept in public repositories), copy the OSCALERT/config/template-unpublished.config-template file to an appropriate subfolder of OSCALERT/config, rename it to unpublished.config, and revise its contents appropriately. Do not add any unpublished.config file to any public repository.
  2. Copy the appropriate unpublished.config file from within the OSCALERT/config folder tree to the OSCALERT top-level folder.
  3. Symlink the appropriate Web.config file from within the OSCALERT/config folder tree to the OSCALERT top-level folder.
From the Windows command line, the prototype commands would be:

cd OSCALERT
copy config/{appropriate-path}/unpublished.config .
mklink Web.config config/{appropriate-path}/Web.config

4 Give write permission in appropriate folders to the host worker process

Some FromPaper2Web applications include features that need to create files on the filesystem, typically in target folders protected/attachment and/or protected/scratch. For this to work, the ASP.NET Machine Account (ASPNET) [on IIS5] or the NETWORK SERVICE account [on IIS7] must have write permission for the target folder(s). Configure this on the Security tab of the folder’s Properties. If the Security tab is missing, open Windows Explorer / Tools / Folder Options... / View, and in the Advanced Settings, clear the "Use simple file sharing" checkbox.

5 Establish a dedicated, specially-configured application pool

OSCALERT is aberrant as an ASP.NET Web application because it presents no useful GUI, uses a headless resource-intensive System.Windows.Forms.WebBrowser object in an explicitly-launched separate thread, is very long-running, and manages its own “recycling” when properly set up as a cron job. Because of these peculiarities, you should not allow it to run in an IIS application pool that also hosts other apps. Instead, you should create a separate and distinct IIS application pool just for OSCALERT.
Furthermore, application pools with default settings do not support such long-running applications or, in some environments, the use of the WebBrowser object. Consequently, the recommended Advanced Settings for OSCALERT’s dedicated application pool are:
  • Process Model / Identity: Administrator (this is not known to be the minimally-privileged setting)
  • Process Model / Idle Time-out (minutes): 0
  • Process Model / Shutdown Time Limit (seconds): 86340
  • Recycling / Disable Overlapped Recycle: True
  • Recycling / Regular Time Interval (minutes): 0
  • Recycling / Request Limit: 1

6 Set up cron jobs

Some FromPaper2Web applications include features that support scheduled unattended jobs, such as notifications and report generation, data scrubs, and other periodic tasks. Crontab entries for any such jobs that are part of OSCALERT are located in noninteractive/.crontab.
